### 09:30 ~ 10:00 | API 스펙 확정 (전원)

아래 스펙을 전원이 합의하고 시작한다. **이 스펙이 확정되어야 각자 독립적으로 개발 가능.**

#### 1) 가맹점 → VAN 승인 요청

> 필드명은 ISO 8583 Data Element 기준을 따른다. (DE번호는 주석으로 병기)

```
POST /api/v1/approval/request

Request Body:
{
  "primaryAccountNumber": "4123456789012345",  // DE02 - PAN (카드번호)
  "expirationDate": "2028-12",                 // DE14 - Expiration Date (유효기간)
  "transactionAmount": 50000,                  // DE04 - Transaction Amount (거래금액)
  "cardAcceptorId": "MERCHANT_001",            // DE42 - Card Acceptor ID (가맹점ID)
  "installmentMonths": 0                       // 국내 전용 필드 (할부개월수)
}

Response:
{
  "systemTraceAuditNumber": "APR20260319001",  // DE11 - STAN (거래 고유번호)
  "responseCode": "00",                        // DE39 - Response Code (00: 승인, 그 외: 거절코드)
  "responseMessage": "승인완료",               // 응답 메시지 (한국어)
  "approvedAt": "2026-03-19T10:30:00",         // 승인 일시
  "cardCompany": "SHINHAN"                     // 카드사명
}
```

> **responseCode 주요 값 (ISO 8583 DE39 기준)**
> - `00`: 승인
> - `51`: 잔액 부족
> - `61`: 신용 한도 초과
> - `54`: 유효기간 만료
> - `14`: 유효하지 않은 카드

#### 2) VAN → 카드사 승인 요청

기본제공 카드사 서버의 API 스펙에 맞춰 호출한다.

#### 3) 결제 결과 조회

```
GET /api/v1/approval/{stan}
```

## 핵심 원칙

1. **Happy Path 먼저** — 정상 결제 흐름 하나가 끝까지 동작하는 것이 최우선
2. **API 스펙 준수** — 오전에 합의한 스펙을 임의로 변경하지 않는다. 변경 필요 시 전원 공유
3. **과감히 버리기** — 매입/정산, Saga, 멱등성 등은 시간이 남을 때만. 발표 시 "추후 개선사항"으로 언급
4. **30분 룰** — 30분 이상 막히면 혼자 고민하지 말고 팀에 공유
5. **문서 변경 이력 기록** — 이 문서를 수정할 경우 상단 변경 이력 테이블에 반드시 기록한다.
    - 형식: `@author (담당자명) | 년.월.일.시.분 | 어떤 부분을 수정했는지`
    - 예시: `개발자2 | 2026.03.19.10.30 | VAN → 카드사 API 스펙 추가`
